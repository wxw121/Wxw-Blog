# AI Agent 工程（二十六）：RAG Agent 的 Query Planning

> Query Planning 不是把用户问题改写得更长，而是识别子问题、数据源、过滤条件和证据完成标准。

---

## 你会学到什么

- 把复杂任务拆成可检索子问题。
- 为每个查询选择数据源和 metadata filter。
- 校验查询计划的覆盖、重复和预算。
- 判断何时直接回答、检索或请求补充。

## 它解决什么问题

问题：

```text
客户为什么没有高级报表权限？和上个月相比发生了什么？
```

需要至少三类证据：

1. 当前套餐是否支持高级报表。
2. 当前用户角色是否包含权限。
3. 上个月套餐或角色是否发生变化。

单个向量查询很难同时覆盖。

本模块完整链路仍是：**Query Planning → Retrieval Tool → Citation Verification → Bad Case Debugging**。

## 最小示例

```python
from pydantic import BaseModel, Field
from typing import Literal


class RetrievalQuery(BaseModel):
    query_id: str
    question: str = Field(min_length=3, max_length=200)
    source: Literal["product_docs", "audit_logs", "tickets"]
    filters: dict = Field(default_factory=dict)
    expected_evidence: str


class QueryPlan(BaseModel):
    queries: list[RetrievalQuery] = Field(min_length=1, max_length=5)
```

```json
{
  "queries": [
    {
      "query_id": "q1",
      "question": "高级报表需要什么套餐？",
      "source": "product_docs",
      "filters": {"version": "current"},
      "expected_evidence": "套餐与功能映射"
    },
    {
      "query_id": "q2",
      "question": "客户角色最近一个月如何变化？",
      "source": "audit_logs",
      "filters": {"event_type": "role_change"},
      "expected_evidence": "带时间的角色变更事件"
    }
  ]
}
```

## 工程化版本

### Planner 只使用已注册数据源

模型不能凭空生成 `crm_v2_secret_db`。source 使用枚举，后端映射到真实 Retrieval Tool。

### Query 计划校验

```python
def validate_query_plan(plan: QueryPlan) -> None:
    fingerprints: set[tuple[str, str]] = set()
    for query in plan.queries:
        key = (query.source, normalize(query.question))
        if key in fingerprints:
            raise ValueError("duplicate_query")
        fingerprints.add(key)

        validate_filters(query.source, query.filters)
```

### 先识别缺失输入

如果用户没有提供 customer_id，Planner 不应该用语义检索猜客户，而应返回 `need_user_input`。

### 查询预算

```text
最多 5 个子查询
每个 source 最多 2 次
总 top_k 不超过 30
总检索时间不超过 5 秒
```

## 常见失败模式

- 把同义 query 当成不同子问题。
- Planner 生成无效 metadata filter。
- 所有查询都打到同一个向量库。
- 没有 expected_evidence，无法判断是否完成。
- 缺 customer_id 时用相似名称猜测。
- 查询计划包含写操作。

## 什么时候不要这么做

问题单一、已知数据源时直接检索。

用户输入缺失时优先澄清，不要生成更多查询掩盖缺口。

数据源无法提供目标证据时，不要让 Planner 反复改写。

## 生产环境注意事项

- source、filter 字段和允许值由后端提供。
- 用户权限决定可用数据源。
- query plan 保存 planner_version。
- 计划变更记录原因。
- 对日期、租户和资源 ID 使用确定性过滤。

## 如何观测和评测

评测：

- 子问题覆盖率。
- 无效数据源率。
- 重复查询率。
- filter 合法率。
- 单任务平均查询数。
- 查询计划相对单查询的 Recall 提升。

Bad Case Debugging 时先看计划是否正确，再看 Retrieval Tool。

## 和 RAG / 后端 / 前端的关系

- RAG 提供 source 能力和 filter schema。
- 后端校验计划和执行预算。
- 前端在输入不足时展示澄清问题。
- Citation Verification 使用 query_id 追踪证据来源。

## 面试怎么讲

> Query Planning 输出子问题、数据源、过滤条件和 expected_evidence。source 和 filter 都来自后端 schema，计划要去重、限数量并检查缺失输入。只有证据目标不同才拆查询；问题单一时直接普通 RAG。

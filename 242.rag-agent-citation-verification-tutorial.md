# AI Agent 工程（二十九）：RAG Agent 引用验证

> 答案带有 [1] 不代表引用正确。Citation Verification 要检查关键事实、引用身份和证据内容是否一致。

---

## 你会学到什么

- 区分引用存在、引用可定位和引用支持。
- 把答案拆成可验证 claim。
- 处理多证据、冲突证据和实时状态。
- 在验证失败时拒绝发布或重新检索。

## 它解决什么问题

常见假引用：

- 引用文档真实存在，但不支持该结论。
- 引用旧版本制度。
- 数字来自业务工具，答案却引用政策文档。
- 一个引用只支持半句话。
- Agent 拼接了不存在的 evidence_id。

这一步位于 **Query Planning、Retrieval Tool** 之后，并把失败送入 **Bad Case Debugging**；核心组件就是 **Citation Verification**。

## 最小示例

```python
from pydantic import BaseModel, Field


class Claim(BaseModel):
    text: str
    evidence_ids: list[str] = Field(min_length=1)


class VerificationResult(BaseModel):
    supported: bool
    unsupported_claims: list[str] = Field(default_factory=list)
    missing_evidence_ids: list[str] = Field(default_factory=list)
```

答案先结构化：

```json
{
  "claims": [
    {
      "text": "一线城市住宿上限为 500 元/晚。",
      "evidence_ids": ["policy-17"]
    }
  ]
}
```

编排器确认 `policy-17` 确实存在于当前 Evidence Store。

## 工程化版本

四层验证：

| 层级 | 检查 |
|---|---|
| Identity | evidence_id 是否属于当前任务 |
| Location | source/page/version 是否可定位 |
| Entailment | 证据是否支持 claim |
| Freshness | 是否使用正确版本或当前状态 |

```python
def verify_identity(
    claims: list[Claim],
    evidence_store: dict[str, Evidence],
) -> list[str]:
    missing: list[str] = []
    for claim in claims:
        for evidence_id in claim.evidence_ids:
            if evidence_id not in evidence_store:
                missing.append(evidence_id)
    return sorted(set(missing))
```

语义支持可以用规则、模型评审或人工抽检，但身份和版本必须确定性校验。

### 业务状态引用

```text
[制度 1] 退款期限为支付后 7 天。
[订单状态 2] 当前订单已支付 10 天。
```

结论需要同时引用规则和状态，不能只引用其中一个。

## 常见失败模式

- 只检查答案有没有引用符号。
- 模型自由生成 evidence_id。
- Evidence Store 混入其他用户任务证据。
- 版本和生效日期未验证。
- Reviewer 只看摘要，不看原证据。
- 引用验证失败后删除引用但保留结论。

## 什么时候不要这么做

纯创意写作不需要事实引用验证。

但只要答案包含政策、金额、状态或操作建议，就应该验证关键 claim。

验证器没有真实 Evidence 时不能自行搜索替代并静默通过；应返回失败给编排器。

## 生产环境注意事项

- evidence_id 由系统生成。
- 引用只允许来自当前 task_id 的 Evidence Store。
- 记录 document_version 和 observed_at。
- 验证失败时重新检索、降级或转人工。
- 用户点击引用时跳到实际来源位置。

## 如何观测和评测

指标：

- Citation presence。
- Evidence identity valid rate。
- Claim support rate。
- Version correctness。
- Unsupported claim rate。
- 点击后来源可访问率。

Bad Case Debugging 保存 claim、evidence_id、验证结果和修复动作。

## 和 RAG / 后端 / 前端的关系

- Retrieval Tool 负责提供可定位 Evidence。
- 后端 Evidence Store 防止引用伪造。
- 前端把引用卡片定位到 source/page。
- Agent 只在验证通过后生成最终发布版本。

## 面试怎么讲

> Citation Verification 不只是检查 [1]。我会先把答案拆成 claim，再验证 evidence_id 属于当前任务、来源可定位、版本正确且内容支持 claim。身份和版本用确定性代码，语义支持可用模型或人工。失败时重新检索或降级，不能只删除引用。

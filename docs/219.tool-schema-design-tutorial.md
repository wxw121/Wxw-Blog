# AI Agent 工程（六）：Tool Schema 设计

> 好工具不是“把现有 API 原样暴露给模型”，而是把一个明确业务动作压缩成模型容易选择、后端容易校验的契约。

---

## 你会学到什么

- 设计稳定的工具名称和描述。
- 控制 schema 粒度，避免万能工具。
- 使用枚举、范围和互斥字段表达约束。
- 判断什么时候应该拆分或合并工具。

## 它解决什么问题

模型通过名称、描述和参数 schema 判断工具是否适合当前任务。如果 schema 模糊，后续再强的模型也会频繁误选。

坏例子：

```json
{
  "name": "operate",
  "description": "处理业务",
  "parameters": {
    "data": {}
  }
}
```

它没有回答：

- 处理什么业务？
- 是读还是写？
- 参数有哪些？
- 什么场景不能使用？
- 失败如何解释？

## 最小示例

```python
from typing import Literal
from pydantic import BaseModel, Field


class TicketSearchInput(BaseModel):
    customer_id: str = Field(
        min_length=1,
        max_length=64,
        description="当前租户内的客户 ID",
    )
    status: Literal["open", "pending", "closed"] | None = Field(
        default=None,
        description="不传时查询全部状态",
    )
    limit: int = Field(default=10, ge=1, le=50)
```

对应工具描述：

```text
名称：search_customer_tickets
用途：查询当前客户的历史工单，用于排障和了解已处理问题。
限制：只读；不能创建或关闭工单；必须先获得 customer_id。
```

描述中的“不能做什么”能减少误调用。

## 工程化版本

### 命名使用“动词 + 业务对象”

推荐：

```text
search_knowledge_base
get_customer_plan
list_customer_tickets
create_support_ticket
```

避免：

```text
handle
process
database
request
tool_1
```

### 一个工具只做一个风险等级的动作

不要设计：

```json
{
  "name": "manage_ticket",
  "arguments": {
    "action": "search | create | close | delete"
  }
}
```

更好的做法是拆成：

```text
search_tickets      risk=read
create_ticket       risk=write
close_ticket        risk=write
delete_ticket       risk=critical
```

这样可以分别授权、审批和审计。

### 用 schema 表达确定性约束

```python
from datetime import date
from pydantic import BaseModel, Field, model_validator


class LogQueryInput(BaseModel):
    service: str = Field(pattern=r"^[a-z][a-z0-9-]{1,40}$")
    start_date: date
    end_date: date
    keyword: str | None = Field(default=None, max_length=100)
    limit: int = Field(default=100, ge=1, le=500)

    @model_validator(mode="after")
    def validate_range(self) -> "LogQueryInput":
        if self.end_date < self.start_date:
            raise ValueError("end_date must not be before start_date")
        if (self.end_date - self.start_date).days > 7:
            raise ValueError("date range must not exceed 7 days")
        return self
```

### Schema 不等于业务权限

`customer_id` 格式合法，不代表当前用户能访问这个客户。schema 负责形状，Policy 负责权限。

## 常见失败模式

- 参数全部可选，模型不知道最小调用条件。
- 枚举用自由文本，出现同义词和拼写差异。
- 描述写实现细节，不写业务用途。
- 参数层级过深，模型经常漏字段。
- 用一个万能工具承载多个风险等级。
- schema 和真实 handler 已经漂移。

## 什么时候不要这么做

不要把每个内部函数都变成工具。只有需要模型动态选择的业务能力才需要工具 schema。

如果动作由 Workflow 固定触发，应由 Workflow 直接调用 handler，复用相同输入模型即可，不必让模型重新选择。

## 生产环境注意事项

工具 schema 应版本化。破坏性变更不要直接修改旧 schema：

```text
search_customer_tickets_v1
search_customer_tickets_v2
```

但也不要长期保留大量版本。迁移完成后，应从 Agent 可见工具列表移除旧版本。

工具描述需要纳入代码评审，因为它会影响模型行为。建议为 schema 建立快照测试，防止字段意外变化。

## 如何观测和评测

为每个工具维护选择测试集：

```json
{
  "user_task": "看看客户 C-18 最近有没有未关闭工单",
  "expected_tool": "search_customer_tickets",
  "expected_arguments": {
    "customer_id": "C-18",
    "status": "open"
  }
}
```

评测：

- 工具选择准确率。
- 必填字段完整率。
- 枚举值合法率。
- 不该调用时的拒绝率。
- schema 版本变更后的回归率。

## 和 RAG / 后端 / 前端的关系

- RAG 工具 schema 应限制 top_k、过滤条件和查询长度。
- 后端用同一个输入模型完成校验，避免模型 schema 与 handler 漂移。
- 前端审批界面根据 schema 生成人可读参数摘要，但高风险字段需要专门文案。

## 面试怎么讲

> 我会把工具设计成单一业务动作，名称使用动词加对象，描述同时写用途和限制。确定性约束进入 JSON Schema 或 Pydantic，业务权限由 Policy 单独校验。读写动作拆成不同工具，避免万能 action 参数混合风险等级。

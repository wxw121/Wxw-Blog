# AI Agent 工程（八）：工具结果标准化

> 工具输入需要 schema，输出同样需要契约。否则每个工具都返回不同形状，模型难以理解，前端难以展示，日志也无法聚合。

---

## 你会学到什么

- 设计统一 ToolResult。
- 区分模型 observation、前端展示和内部原始数据。
- 控制结果长度并保留证据引用。
- 用稳定错误码支持重试和降级。

## 它解决什么问题

未经设计的工具可能返回：

```text
工具 A：一段字符串
工具 B：数据库行数组
工具 C：第三方完整 HTTP 响应
工具 D：异常堆栈
```

编排器不得把这些原样交给模型。需要先转换成统一结果：

```json
{
  "ok": true,
  "data": {
    "items": []
  },
  "error": null,
  "trace_id": "tool-call-001"
}
```

## 最小示例

```python
from typing import Any, Generic, TypeVar
from pydantic import BaseModel, Field


T = TypeVar("T")


class ToolError(BaseModel):
    code: str
    message: str
    retryable: bool = False


class ToolResult(BaseModel, Generic[T]):
    ok: bool
    data: T | None = None
    error: ToolError | None = None
    trace_id: str
    warnings: list[str] = Field(default_factory=list)


def success(data: Any, trace_id: str) -> ToolResult[Any]:
    return ToolResult(ok=True, data=data, trace_id=trace_id)


def failure(
    code: str,
    message: str,
    trace_id: str,
    *,
    retryable: bool = False,
) -> ToolResult[None]:
    return ToolResult(
        ok=False,
        error=ToolError(
            code=code,
            message=message,
            retryable=retryable,
        ),
        trace_id=trace_id,
    )
```

## 工程化版本

同一原始结果通常要生成三个视图：

| 视图 | 用途 | 内容 |
|---|---|---|
| Internal | 调试与业务处理 | 完整但受访问控制的原始数据 |
| Model | 下一步决策 | 裁剪、脱敏、带证据 ID |
| UI | 用户展示 | 人类可读摘要和状态 |

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class NormalizedToolOutput:
    internal: dict
    model_observation: dict
    ui_summary: dict


def normalize_ticket_search(raw: list[dict]) -> NormalizedToolOutput:
    model_items = [
        {
            "ticket_id": row["id"],
            "status": row["status"],
            "summary": row["summary"][:300],
        }
        for row in raw[:20]
    ]

    return NormalizedToolOutput(
        internal={"items": raw},
        model_observation={"items": model_items, "truncated": len(raw) > 20},
        ui_summary={"count": len(raw), "shown": len(model_items)},
    )
```

### 保留证据身份

RAG 工具输出至少包含：

```json
{
  "chunk_id": "policy-17",
  "source": "travel-policy.pdf",
  "page": 8,
  "text": "一线城市住宿上限为 500 元/晚。",
  "score": 0.87
}
```

这样后续答案才能引用并验证来源。

### 长结果分页

工具不要一次返回全部数据。可以返回 `next_cursor`，由 Agent 判断是否需要继续。

```json
{
  "items": [],
  "next_cursor": "ticket:20",
  "has_more": true
}
```

## 常见失败模式

- 成功和失败使用完全不同顶层结构。
- `ok=false` 但没有稳定错误码。
- 把第三方错误堆栈直接返回模型。
- 工具返回全文，迅速耗尽上下文。
- 裁剪结果时丢失 chunk_id 或业务主键。
- 模型 observation 和用户 UI 共用敏感原始字段。

## 什么时候不要这么做

不要为了统一而丢失工具特有语义。统一的是顶层 envelope、错误和追踪字段，`data` 内部仍应使用领域类型。

普通内部函数如果不经过 Agent、前端或跨服务边界，可以直接返回领域对象，不必强制包装 ToolResult。

## 生产环境注意事项

标准化层应设置：

- 最大 item 数。
- 单字段最大长度。
- 总 observation 大小。
- 敏感字段黑名单和白名单。
- 二进制内容的对象存储引用。
- 时间、金额和枚举的统一格式。

如果结果被截断，必须显式返回 `truncated=true`，不能让模型误以为数据完整。

## 如何观测和评测

监控：

- 工具原始结果大小。
- observation 压缩比例。
- 结果截断率。
- 错误码分布。
- 模型因结果不完整发起的重复调用。

建立快照测试，保证同一 handler 版本返回的顶层结构稳定。

## 和 RAG / 后端 / 前端的关系

- RAG 结果必须保留来源身份和检索分数。
- 后端负责把原始领域对象转换成三种视图。
- 前端使用 `ui_summary` 展示进度和结果卡片。
- 模型只接收完成下一步所需的最小 observation。

## 面试怎么讲

> 我会统一工具结果 envelope：ok、data、error、trace_id 和 warnings，但 data 保持领域类型。原始结果、模型 observation 和前端摘要分开，模型侧要裁剪脱敏并保留证据 ID。错误使用稳定 code 和 retryable 标志支持编排器决策。

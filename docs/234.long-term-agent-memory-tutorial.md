# AI Agent 工程（二十一）：长期记忆

> 长期记忆让 Agent 跨会话复用稳定偏好和历史经验，但它也带来隐私、错误固化和用户控制问题。

---

## 你会学到什么

- 判断什么信息值得跨会话保存。
- 设计带来源、置信度和过期时间的 Memory。
- 区分显式记忆和推断记忆。
- 防止错误与敏感信息长期固化。

## 它解决什么问题

合理长期记忆：

- 用户明确要求以后默认中文。
- 用户负责的项目代号是 Atlas。
- 用户偏好答案先结论后解释。
- 过去任务中已经确认的工具使用限制。

不合理长期记忆：

- 模型猜测用户是管理员。
- 一次临时请求“今天回答简短点”。
- 完整身份证号或访问 token。
- 已经过期的项目角色。
- 组织制度正文。

## 最小示例

```python
from datetime import datetime
from pydantic import BaseModel, Field
from typing import Literal


class MemoryRecord(BaseModel):
    memory_id: str
    user_id: str
    tenant_id: str
    category: Literal["preference", "profile", "experience"]
    content: str = Field(max_length=500)
    source: Literal["user_explicit", "user_confirmed", "system_event"]
    confidence: float = Field(ge=0, le=1)
    created_at: datetime
    expires_at: datetime | None = None
    deleted_at: datetime | None = None
```

只有 `user_explicit`、`user_confirmed` 或可信系统事件可以进入长期记忆。

## 工程化版本

写入策略：

| 信息类型 | 是否写入记忆 | 原因 | 删除策略 |
|---|---|---|---|
| 用户明确语言偏好 | 是 | 稳定且提升体验 | 用户删除或偏好变更 |
| 用户确认的项目别名 | 是，可过期 | 跨会话有帮助 | 项目结束自动过期 |
| 模型推测的职位 | 否 | 未经验证 | 不保存 |
| 一次任务中的订单号 | 否 | 只属于任务状态 | 任务保留期结束 |
| API token | 否 | 高敏感 Secret | 不进入 Memory |

### 显式写入优先

用户说“以后都用中文回答”可以直接生成候选：

```json
{
  "category": "preference",
  "content": "默认使用中文回答",
  "source": "user_explicit",
  "confidence": 1.0
}
```

### 更新而不是无限追加

用户从“简洁回答”改成“需要详细解释”时，应更新同一偏好类别，而不是保留冲突记录同时召回。

### 检索先过滤后排序

```python
def retrieve_memories(user: UserContext, query: str) -> list[MemoryRecord]:
    candidates = memory_store.search(
        user_id=user.user_id,
        tenant_id=user.tenant_id,
        query=query,
        include_deleted=False,
    )
    return [
        item for item in candidates
        if item.expires_at is None or item.expires_at > now()
    ][:5]
```

## 常见失败模式

- 自动保存每轮对话。
- 同一个偏好存在互相冲突的多条记录。
- 没有 source，无法知道记忆从哪里来。
- 过期记忆仍参与决策。
- 使用共享向量索引但没有用户过滤。
- 用户无法查看和纠正。

## 什么时候不要这么做

没有明确收益时不要增加长期记忆。很多客服、检索和一次性分析任务只需要会话状态。

敏感、法律、医疗或财务推断不应因为“个性化”自动保存。

组织事实应进入 RAG 文档治理，不进入个人 Memory。

## 生产环境注意事项

- 用户能查看、编辑、删除长期记忆。
- 每条记录带来源和过期时间。
- 写入前做 PII 和 Secret 检测。
- 删除覆盖主存储、向量索引、缓存和备份策略。
- Memory 不能提升用户权限。
- 把召回 Memory 标记为历史偏好，不当作当前事实。

## 如何观测和评测

指标：

- 每用户平均 Memory 数。
- 候选写入接受率。
- Memory 召回后使用率。
- 用户纠正率和删除率。
- 过期记录拦截数。
- 个性化质量提升与错误率。

离线评测要包含偏好变更、冲突记录、过期项目和跨用户隔离。

## 和 RAG / 后端 / 前端的关系

- RAG 管理组织知识，长期 Memory 管理用户相关信息。
- 后端实现写入 Policy、过滤和删除。
- 前端提供“已记住的信息”管理页。
- Agent 使用 Memory 时应在必要场景让用户知道。

## 面试怎么讲

> 长期记忆只保存经用户明确表达或确认、跨会话有价值的信息。记录带 user/tenant、source、confidence、expires_at 和删除状态。检索先做身份和过期过滤，再做语义排序；用户可以查看、纠正和删除。组织制度和任务状态不进入长期 Memory。

# AI Agent 工程（七）：工具参数校验

> 模型输出通过 JSON 解析，只能说明格式像 JSON；通过 schema，只能说明字段形状正确。真正可执行前，还需要业务、权限和状态校验。

---

## 你会学到什么

- 建立四层工具参数校验。
- 区分语法、schema、业务和权限错误。
- 设计模型可理解、用户可处理的错误。
- 防止路径注入、范围越界和跨租户访问。

## 它解决什么问题

下面的参数格式完全合法，但仍可能危险：

```json
{
  "customer_id": "C-999",
  "refund_amount": 1000000,
  "reason": "模型认为应该退款"
}
```

可能的问题：

- C-999 不属于当前租户。
- 退款金额超过订单实付。
- 订单已经退款。
- 当前用户没有退款权限。
- 金额超过自动处理阈值，需要人工确认。

## 最小示例

```python
from decimal import Decimal
from pydantic import BaseModel, Field


class RefundInput(BaseModel):
    order_id: str = Field(pattern=r"^ORD-[0-9]{4,12}$")
    amount: Decimal = Field(gt=0, max_digits=10, decimal_places=2)
    reason: str = Field(min_length=5, max_length=300)


def validate_refund(
    value: RefundInput,
    *,
    paid_amount: Decimal,
    already_refunded: Decimal,
    can_refund: bool,
) -> None:
    if not can_refund:
        raise PermissionError("refund_permission_required")

    refundable = paid_amount - already_refunded
    if value.amount > refundable:
        raise ValueError("amount_exceeds_refundable_balance")
```

schema 负责格式和数值范围，`validate_refund` 负责业务状态和权限。

## 工程化版本

推荐四层校验：

| 层级 | 检查内容 | 失败示例 |
|---|---|---|
| 1. 解析 | 是否为合法 JSON | 缺少引号 |
| 2. Schema | 类型、必填、长度、枚举 | top_k 是字符串 |
| 3. 业务 | 资源状态、金额、时间范围 | 已关闭工单不能再次关闭 |
| 4. 权限 | 用户、租户、角色、字段范围 | 跨租户 customer_id |

```python
def prepare_tool_call(
    raw_arguments: dict,
    *,
    input_model: type[BaseModel],
    user: UserContext,
) -> BaseModel:
    value = input_model.model_validate(raw_arguments)
    validate_tenant_scope(value, user.tenant_id)
    validate_business_rules(value)
    validate_permissions(value, user.permissions)
    return value
```

### 路径和查询校验

文件工具不能接受任意绝对路径：

```python
from pathlib import Path


def safe_workspace_path(root: Path, relative: str) -> Path:
    candidate = (root / relative).resolve()
    root = root.resolve()
    if root not in candidate.parents and candidate != root:
        raise ValueError("path_outside_workspace")
    return candidate
```

数据库工具不能让模型直接生成 SQL。更稳妥的是暴露业务参数，由后端构建参数化查询。

### 错误分类

```json
{
  "ok": false,
  "error": {
    "code": "amount_exceeds_refundable_balance",
    "message": "退款金额超过可退余额",
    "retryable": false,
    "field": "amount"
  }
}
```

模型可以根据 `retryable` 判断是否应该修正参数；用户可以根据 `message` 理解问题；内部堆栈不应返回。

## 常见失败模式

- 只做 Pydantic 校验，不做租户和业务状态校验。
- 把用户不可见的内部 ID 交给模型猜。
- 错误只返回 `invalid arguments`，无法修正。
- 错误里包含 SQL、文件路径或堆栈。
- 校验失败后无限让模型重试。
- 写工具在校验和执行之间没有重新检查资源版本。

## 什么时候不要这么做

如果工具参数可以由后端从已认证上下文确定，就不要让模型提供。例如 `tenant_id`、`user_id` 和权限集合应该由后端注入。

如果参数依赖强一致业务状态，应该在事务内由业务服务校验，而不是只在 Agent 编排层校验。

## 生产环境注意事项

防止 TOCTOU（检查时和使用时状态不同）：

```text
Agent 校验订单可退款
  → 等待人工确认 5 分钟
  → 订单期间已经退款
  → 用户确认
```

真正执行时必须再次校验订单版本和可退金额。批准对象应带资源版本：

```json
{
  "order_id": "ORD-1001",
  "order_version": 17,
  "amount": "680.00"
}
```

## 如何观测和评测

记录校验失败阶段和错误码：

- parse_failure。
- schema_failure。
- business_rule_failure。
- permission_denied。
- stale_resource。

评测集应包含边界值、跨租户 ID、过期资源、重复请求和恶意路径。

## 和 RAG / 后端 / 前端的关系

- RAG 工具要限制 query 长度、top_k 和 metadata filter。
- 后端业务服务是最终规则来源。
- 前端确认页应展示经过校验的参数，而不是原始模型文本。
- 校验错误需要转成人可理解提示和模型可处理错误码。

## 面试怎么讲

> 工具参数校验分四层：解析、schema、业务规则和权限。tenant_id、user_id 等上下文由后端注入，不能让模型提供。高风险动作在人工确认后还要做执行前二次校验，避免资源状态已经变化。

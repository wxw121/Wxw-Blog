# AI Agent 工程（十七）：复杂任务拆解

> 好的拆解不是把一句话变成更多句子，而是把目标变成有明确依赖、输入、输出和完成条件的可执行子任务。

---

## 你会学到什么

- 识别任务中的数据依赖和动作依赖。
- 生成可验证的子任务。
- 控制拆解粒度和总步骤。
- 区分并行步骤与顺序步骤。

## 它解决什么问题

目标：

```text
分析本季度客户投诉，找出前三类原因，
对照服务制度给出改进建议，并创建复盘任务。
```

直接让 Agent 执行容易遗漏。合理拆解：

1. 获取本季度投诉。
2. 聚类并统计原因。
3. 检索相关制度。
4. 对比问题与制度。
5. 生成建议。
6. 人工确认后创建复盘任务。

## 最小示例

```python
from pydantic import BaseModel, Field


class SubTask(BaseModel):
    id: str
    objective: str
    required_inputs: list[str] = Field(default_factory=list)
    expected_output: str
    completion_check: str
    depends_on: list[str] = Field(default_factory=list)


class TaskGraph(BaseModel):
    goal: str
    tasks: list[SubTask] = Field(min_length=1, max_length=10)
```

一个子任务示例：

```json
{
  "id": "s3",
  "objective": "检索与前三类投诉相关的服务制度",
  "required_inputs": ["s2.top_categories"],
  "expected_output": "带 source/page 的制度证据列表",
  "completion_check": "每类至少一条可定位证据",
  "depends_on": ["s2"]
}
```

## 工程化版本

### 先识别任务类型

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 数据获取 | 查询投诉 | 工具 |
| 语义分析 | 归类原因 | 模型或分析服务 |
| 知识检索 | 查制度 | RAG |
| 确定计算 | 统计占比 | 代码 |
| 写操作 | 创建任务 | 工具 + 审批 |

### 拆解粒度

每个子任务最好满足：

- 单一目标。
- 明确输入。
- 结构化输出。
- 可检查完成条件。
- 一个主要失败原因。

### 校验依赖图

```python
def validate_graph(graph: TaskGraph) -> None:
    ids = {task.id for task in graph.tasks}
    if len(ids) != len(graph.tasks):
        raise ValueError("duplicate_task_id")

    for task in graph.tasks:
        missing = set(task.depends_on) - ids
        if missing:
            raise ValueError(f"missing_dependencies:{sorted(missing)}")

    ensure_acyclic(graph.tasks)
```

### 并行只用于无依赖只读步骤

查询三套互不依赖的资料可以并行；共享写资源或依赖前一步结果时必须串行。

## 常见失败模式

- 子任务只是原问题的同义改写。
- 没有 expected_output 和 completion_check。
- 拆得过细，模型调用和状态管理成本暴涨。
- 写操作与分析步骤混在一起。
- 忽略依赖，使用不存在的中间结果。
- 用户目标变化后继续执行旧图。

## 什么时候不要这么做

简单任务不需要拆解。

固定业务流程应由 Workflow 预定义，不必每次让模型重新生成 TaskGraph。

如果无法验证子任务完成条件，应先要求用户补充目标或转人工。

## 生产环境注意事项

- 限制最大子任务数和最大深度。
- 计划中只允许已注册任务类型。
- 写任务默认 blocked。
- 保存中间产物 schema，不用纯文本串联所有步骤。
- 用户修改目标时创建新 graph_version。

## 如何观测和评测

指标：

- TaskGraph 校验通过率。
- 平均子任务数和深度。
- 无用子任务比例。
- 依赖错误率。
- 子任务完成率。
- Replan 触发原因。

评测不仅看最终结果，还要判断拆解是否最小、完整且无循环。

## 和 RAG / 后端 / 前端的关系

- RAG 子任务必须输出带来源证据。
- 后端保存 TaskGraph 和中间产物。
- 前端展示任务进度，但避免暴露敏感内部参数。
- Workflow 可以直接执行通过校验的图。

## 面试怎么讲

> 任务拆解要生成有 objective、required_inputs、expected_output、completion_check 和 depends_on 的结构化子任务，不是自然语言清单。我会限制数量和深度、检查 DAG，并把写操作标记为 blocked。固定流程不动态拆解，直接使用 Workflow。

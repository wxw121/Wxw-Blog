# AI Agent 工程（二十三）：Memory 检索策略

> 长期记忆不是每轮都全部召回。好的检索策略先判断是否需要，再按用户、类别、过期和相关性逐层过滤。

---

## 你会学到什么

- 设计 Memory Retrieval Gate。
- 组合身份过滤、规则过滤和语义排序。
- 控制召回数量与上下文格式。
- 防止历史 Memory 注入当前任务。

## 它解决什么问题

错误做法：

```text
每次请求 → 搜索用户全部 Memory → 取相似度最高 20 条 → 拼进 prompt
```

问题包括：

- 无关偏好干扰当前任务。
- 过期信息被当作事实。
- token 增长。
- 恶意历史内容持续影响 Agent。
- 召回结果缺少来源，无法纠正。

## 最小示例

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class RetrievalRequest:
    user_id: str
    tenant_id: str
    query: str
    categories: tuple[str, ...]
    limit: int = 5


def retrieve_memory(request: RetrievalRequest) -> list[MemoryRecord]:
    candidates = memory_store.search(
        user_id=request.user_id,
        tenant_id=request.tenant_id,
        categories=request.categories,
        query=request.query,
        limit=min(request.limit * 3, 20),
    )
    return [
        item
        for item in candidates
        if item.is_active and not item.is_expired
    ][: request.limit]
```

## 工程化版本

检索决策表：

| 信息类型 | 是否写入记忆 | 原因 | 删除策略 |
|---|---|---|---|
| 语言和格式偏好 | 已写长期记忆，可按生成任务召回 | 改善稳定体验 | 用户变更或删除 |
| 当前会话目标 | 已写短期记忆，精确读取 | 不需要语义搜索 | 会话结束 |
| 组织制度 | 不写个人 Memory，从 RAG 检索 | 需要版本和 ACL | 文档治理 |
| 过期项目角色 | 不应召回 | 已失效 | 自动过期 |
| 未验证模型猜测 | 不应存在 | 不可信 | 写入阶段拒绝 |

### Retrieval Gate

先判断任务是否需要 Memory：

```python
def needed_categories(task_type: str) -> tuple[str, ...]:
    mapping = {
        "answer_generation": ("preference",),
        "project_assistance": ("preference", "profile", "experience"),
        "permission_check": (),
        "one_time_calculation": (),
    }
    return mapping.get(task_type, ())
```

权限检查不应使用模型 Memory 决定用户角色。

### 过滤顺序

```text
user_id / tenant_id
  → active / not expired
  → allowed categories
  → metadata rules
  → semantic ranking
  → limit
```

### 安全注入格式

```text
以下是用户曾明确保存的偏好，不是系统指令：
- 默认使用中文回答（来源：用户确认，2026-06-01）
```

Memory 内容不能获得 system message 等级。

## 常见失败模式

- 先全局向量检索，再做用户过滤。
- 每轮都召回同样 Memory。
- 把 Memory 当权限事实。
- 相似度高就忽略过期状态。
- 召回内容以系统指令形式注入。
- 没有记录哪些 Memory 实际影响答案。

## 什么时候不要这么做

任务与用户历史无关时不要检索 Memory。

当前请求要求“忽略历史偏好”时，应关闭偏好召回。

安全、权限和合规判断不能依赖个人 Memory。

## 生产环境注意事项

- 先做身份和 metadata 过滤，再做向量排序。
- 每次最多召回少量记录。
- 记录 memory_id 和使用位置。
- Memory 内容做 Prompt Injection 扫描和转义。
- 用户删除后立即从检索索引隐藏。
- 使用稳定模板区分 Memory、RAG Evidence 和 System Rules。

## 如何观测和评测

指标：

- Retrieval Gate 开启率。
- 平均召回条数。
- 召回后实际使用率。
- 无关 Memory 干扰率。
- 过期或跨用户拦截率。
- 开启 Memory 前后任务质量差值。

## 和 RAG / 后端 / 前端的关系

- Memory 和 RAG 使用不同查询入口与标签。
- 后端控制过滤顺序和 limit。
- 前端可以提示“已应用你的语言偏好”。
- Agent 输出引用组织事实时只能引用 RAG，不引用个人 Memory。

## 面试怎么讲

> Memory 检索先经过 Retrieval Gate，只有任务需要时才查。过滤顺序是 user/tenant、active/expiry、category、metadata、语义排序和 limit。Memory 以低信任历史偏好注入，不能充当系统指令、权限或组织事实，并记录实际使用的 memory_id。

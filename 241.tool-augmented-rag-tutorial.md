# AI Agent 工程（二十八）：Tool-Augmented RAG

> 文档告诉系统“规则是什么”，业务工具告诉系统“当前发生了什么”。Tool-Augmented RAG 把静态知识证据和实时业务状态组合起来。

---

## 你会学到什么

- 区分知识工具和业务工具。
- 把不同来源结果统一成 Evidence。
- 控制工具顺序、权限和新鲜度。
- 避免用文档猜实时状态。

## 它解决什么问题

用户问：

```text
“为什么我不能申请退款？”
```

需要：

- RAG：退款制度和适用条件。
- 订单工具：订单金额、支付状态、可退余额。
- 工单工具：是否存在处理中的退款工单。

只查制度无法知道当前订单状态，只查订单又无法解释规则。

完整链路保持：**Query Planning、Retrieval Tool、Citation Verification、Bad Case Debugging**，业务工具作为额外 Evidence Provider。

## 最小示例

```python
from dataclasses import dataclass
from typing import Literal


@dataclass(frozen=True)
class Evidence:
    evidence_id: str
    kind: Literal["document", "business_state"]
    source: str
    content: dict
    observed_at: str


def collect_refund_evidence(order_id: str) -> list[Evidence]:
    policy = search_refund_policy(order_id)
    order = get_order(order_id)
    tickets = list_refund_tickets(order_id)
    return [*policy, order, *tickets]
```

## 工程化版本

### 工具分类

| 类型 | 示例 | 特点 |
|---|---|---|
| Knowledge Tool | search_policy | 文档版本、引用、ACL |
| State Tool | get_order | 实时性、资源权限 |
| Action Tool | create_refund | 写入、幂等、审批 |

前三十篇中，Agentic RAG 优先接 Knowledge 和 State Tool；Action Tool 需要 Human-in-the-loop。

### 统一 Evidence

```json
{
  "evidence_id": "order:ORD-1001:v17",
  "kind": "business_state",
  "source": "order_service",
  "content": {
    "status": "paid",
    "refundable_balance": "0.00"
  },
  "observed_at": "2026-07-22T12:00:00+08:00"
}
```

实时状态必须带 observed_at 或 resource_version，避免缓存旧值。

### 先读后写

```text
查制度 → 查订单 → 生成建议 → 人工确认 → 执行动作
```

模型不能从“建议退款”直接跳到 create_refund。

## 常见失败模式

- 用 RAG 文档推测实时订单状态。
- 业务工具结果没有 observed_at。
- 文档 Evidence 和业务 Evidence 混成无来源文本。
- 查询工具使用系统超级权限。
- Planner 同时触发互相依赖的读写工具。
- 答案引用制度，但关键结论实际来自未展示的业务状态。

## 什么时候不要这么做

只需实时结构化数据时直接调用 API，不需要 RAG。

只需解释制度时普通 RAG 足够。

业务工具权限和幂等还没准备好时，不接写工具。

## 生产环境注意事项

- 所有工具继承用户和租户权限。
- 文档 Evidence 带 version，业务 Evidence 带 observed_at。
- 写动作执行前重新读取关键状态。
- 不把完整业务对象放入模型。
- 证据冲突时优先使用明确版本和时间规则。

## 如何观测和评测

分别评测：

- Query Planning 是否选择正确 Evidence Provider。
- Retrieval Tool 是否找到相关制度。
- State Tool 是否返回当前资源版本。
- Citation Verification 是否区分文档和业务证据。
- Bad Case Debugging 是否能定位旧缓存、错权限或错制度。

## 和 RAG / 后端 / 前端的关系

- RAG 提供规则证据。
- 后端业务工具提供实时状态并执行权限。
- 前端同时展示政策引用和订单状态摘要。
- Agent 负责组合，不负责创造缺失事实。

## 面试怎么讲

> Tool-Augmented RAG 把文档知识和实时业务状态统一成带身份的 Evidence。Knowledge Tool 返回文档版本和引用，State Tool 返回 resource_version 或 observed_at，Action Tool 单独走幂等和审批。最终答案区分规则依据和实时状态，不能用 RAG 猜业务数据。

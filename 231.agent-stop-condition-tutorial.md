# AI Agent 工程（十八）：Agent 停止条件

> Agent 的安全不只取决于“能做什么”，也取决于“什么时候必须停止”。停止条件应该由编排器强制执行，不能只靠模型自觉。

---

## 你会学到什么

- 设计成功、预算、失败、权限和人工五类停止条件。
- 区分停止、暂停和取消。
- 防止无限循环和无进展调用。
- 记录可解释的 stop_reason。

## 它解决什么问题

没有停止条件的 Agent 会：

- 不断重复相同检索。
- 在依赖服务失败时无限重试。
- 没有证据仍继续生成。
- 越过用户权限寻找替代工具。
- 用户取消后仍执行后台动作。

## 最小示例

```python
from dataclasses import dataclass


@dataclass
class StopPolicy:
    max_steps: int = 6
    max_tool_calls: int = 10
    max_consecutive_failures: int = 2
    deadline_ms: int = 30_000


def check_stop(state: AgentState, policy: StopPolicy) -> str | None:
    if state.cancelled:
        return "cancelled"
    if state.completed:
        return "completed"
    if state.step >= policy.max_steps:
        return "max_steps"
    if state.tool_calls >= policy.max_tool_calls:
        return "tool_budget_exhausted"
    if state.consecutive_failures >= policy.max_consecutive_failures:
        return "failure_limit"
    if state.elapsed_ms >= policy.deadline_ms:
        return "deadline"
    return None
```

每轮 Decide 之前和 Act 之后都检查。

## 工程化版本

## 停止条件清单

- 达到最大步骤数。
- 工具连续失败超过阈值。
- 已经得到可验证答案。
- 需要人工确认。
- 当前任务超出权限边界。

还应加入：

- 用户取消。
- 总时间或 token 预算耗尽。
- 检测到重复调用和无进展。
- 依赖数据不足，需要用户补充。
- 工具返回不可恢复冲突。

### 停止、暂停和取消

| 状态 | 含义 | 能否恢复 |
|---|---|---|
| completed | 目标完成 | 不需要 |
| failed | 不可恢复失败 | 通常新建任务 |
| waiting_for_approval | 等待确认 | 可以 |
| waiting_for_user_input | 等待补充 | 可以 |
| cancelled | 用户取消 | 由业务决定 |
| max_steps | 预算耗尽 | 人工复核后决定 |

### 无进展检测

```python
def repeated_fingerprint(steps: list[Step]) -> bool:
    recent = [step.fingerprint for step in steps[-3:]]
    return len(recent) >= 2 and len(set(recent)) == 1
```

fingerprint 可由 tool_name、规范化参数和结果摘要组成。

### 完成必须可验证

模型说“完成了”不够。编排器检查：

- 必需产物是否存在。
- 引用是否可定位。
- 写动作是否真正返回成功状态。
- 用户要求的字段是否齐全。

## 常见失败模式

- 最大步数写在 prompt，后端不计数。
- 暂停被当成完成。
- 失败后生成一个自信答案掩盖错误。
- 任务恢复时预算重置。
- 用户取消只关闭前端连接，后台继续。
- 达到 deadline 后仍等待工具返回并执行写入。

## 什么时候不要这么做

不要让模型自由定义硬安全上限。模型可以建议任务完成，但 max_steps、deadline、权限和取消由编排器决定。

也不要把所有失败都标成 completed with warning；这会污染任务完成率。

## 生产环境注意事项

- 停止检查使用单调时钟计算 elapsed time。
- 每次恢复读取原预算。
- 取消信号传播到正在运行的工具。
- 写操作开始后取消要定义补偿策略。
- stop_reason 使用稳定枚举，不使用自由文本。
- 最终答案明确区分完成、部分完成和失败。

## 如何观测和评测

统计 stop_reason 分布：

```json
{
  "completed": 812,
  "waiting_for_approval": 91,
  "max_steps": 37,
  "failure_limit": 24,
  "permission_denied": 18,
  "cancelled": 12
}
```

`max_steps` 或重复调用持续上升，通常表示工具描述、计划或完成检查有问题。

## 和 RAG / 后端 / 前端的关系

- RAG 无结果时应停止或请求补充，而不是编造。
- 后端强制硬停止条件。
- 前端允许取消并展示停止原因。
- Workflow 把 waiting 状态持久化，等待事件后恢复。

## 面试怎么讲

> 停止条件分为完成、预算、失败、权限和人工等待。模型只能建议完成，编排器强制 max_steps、tool budget、deadline、失败阈值和取消。waiting_for_approval 是暂停而不是完成，任务恢复后预算不重置；最终 stop_reason 使用稳定枚举进入 trace。

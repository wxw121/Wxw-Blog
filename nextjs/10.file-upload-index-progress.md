## 1. 前言：没有上传就没有 RAG

第九篇典型延伸：

- 第五篇 `POST` 用 `JSON.stringify`——**传 PDF 还能这样吗？**
- 上传后要等索引，页面怎么显示 **「处理中」**？
- 轮询和 `useEffect` 怎么配合、离开页面要不要停？
- 第六篇说 `/documents` 可用 Server——上传和轮询要不要 **`'use client'`**？

**Ingest**（入库 / 摄取）：把上传文件解析、分块、向量化并写入索引的全过程。  
通俗说：从「一个 PDF」变成「能检索的一堆 chunk」。

**FormData**（表单数据对象）：用 `multipart/form-data` 上传**文件 + 普通字段**。  
通俗说：寄快递的「包裹 + 面单」，不是只寄 JSON 纸条。

读完本文，你应该能做到：

1. 说明为何不能用 `application/json` 塞文件二进制。
2. 用 `FormData` + `fetch` POST，处理上传 loading / error。
3. 实现 FastAPI `UploadFile` + 内存状态机（`pending` → `running` → `done` / `failed`）。
4. 用 `useEffect` 轮询 `GET /api/documents`，卸载时 `clearTimeout`。
5. 在 `/documents` 完成列表 + 进度，并链到 `/chat`。

## 2. 上传在 RAG 链路里站哪

```mermaid
flowchart LR
  Upload["用户上传"]
  Parse["解析分块"]
  Embed["向量化"]
  Index["写入向量库"]
  Chat["问答 + 引用"]
  Upload --> Parse --> Embed --> Index --> Chat
```

读图时看**本篇覆盖**：「选文件」→「任务状态可查」；Parse/Embed 在后端用 `asyncio.sleep` 模拟。

| 阶段 | 用户可见 | 本篇 |
|------|----------|------|
| 选择文件 | 文件名 | §5 |
| 上传中 | 「上传中…」 | §5 |
| 索引中 | `running` + 进度 | §6～§7 |
| 完成 | `done` | §8 → `/chat` |
| 失败 | `failed` | §7 |

### 2.1 和第九篇「模拟文档名」怎么串故事

第九篇 `MOCK_CITATIONS` 里常有 `docs/rag-intro.md`。第十篇上传后，列表里会出现**真实文件名**——演示时可以：

1. 准备一份 `rag-intro.md` 上传；
2. 等 `done`；
3. 去 `/chat` 问「什么是 RAG」；
4. 向听众说明：「引用里的路径和刚上传的文件对应；真后端会检索刚索引的 chunk。」

这样 7～10 篇不再是孤立页面，而是**一条业务时间线**。

### 2.2 BackgroundTasks 与「先 201 再慢慢跑」

用户上传时最糟的体验是：**浏览器转圈 30 秒没响应**。FastAPI 模式是：

```text
POST /api/documents
  → 立刻 201 + { id, status: pending, progress: 0 }
  → HTTP 响应结束，用户已看到列表新行
  → 后台 fake_ingest 慢慢改 progress
```

前端「上传中…」只覆盖 **POST 往返**（通常 <1s）；**0%→100%** 是另一个阶段——别用一个 spinner 盖住全程。

### 2.3 从文件到 chunk：后端黑盒（面试常问）

前端只见 `progress`，但应能口头说出 ingest 在干什么：

```text
文件 bytes → 抽文本 → 切块 chunk → Embedding 向量 → 写入向量库 + 元数据
```

本篇用 `sleep` 模拟耗时；真系统每阶段都可更新 `progress`。第十篇 UI 模式不变。

### 2.4 演示顺序：为何 upload 在 chat 前

| 顺序 | 观众心智 |
|------|----------|
| 先 chat | 知识是 mock 的 |
| upload → done → chat | 知识是我刚灌进去的 |

第十二篇演示脚本把 upload 放第一步，就是在建立因果链。

## 3. FormData 与 multipart：和 JSON POST 有何不同

[第四篇](04.server-actions-post-create.md) / 第五篇用户创建：

```javascript
fetch('/api/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name, email }),
})
```

文件上传**不要** `JSON.stringify` 文件，也**不要**手写 `Content-Type: application/json`。

```javascript
const form = new FormData()
form.append('file', file)

const res = await fetch('/api/documents', {
  method: 'POST',
  body: form,
  // 不设 Content-Type —— 浏览器自动带 multipart boundary
})
```

先错或对：

```javascript
// ❌
body: JSON.stringify({ file: file.name })

// ✅
body: formData
```

| 对比 | JSON POST | 文件上传 |
|------|-----------|----------|
| body | 字符串 | `FormData` |
| Content-Type | `application/json` | `multipart/form-data`（自动） |

Next 仍走第五篇 **`rewrites`**：`fetch('/api/documents')` → FastAPI `:8000`。

### 3.1 用 DevTools 看 multipart 长什么样

上传时在 Network 点开 `documents` 请求：

| 观察项 | 正常 | 异常 |
|--------|------|------|
| Request Headers 的 Content-Type | `multipart/form-data; boundary=----...` | `application/json` |
| Request Payload | 有 `file` 字段与二进制 | 只有 JSON 文本 |
| Response | `201` + `id`、`status: pending` | `422` 多为字段名不对 |

**boundary** 由浏览器生成——你若手写 `Content-Type` 却没有正确 boundary，后端无法切分文件块。

### 3.2 与第四篇「表单三态」对照

| | 创建用户（四） | 上传文档（十） |
|---|----------------|----------------|
| 触发 | 提交表单 | 选文件 + 点上传 |
| body | `JSON.stringify` | `FormData` |
| 成功反馈 | 跳转或列表刷新 | 列表新行 + 后台 ingest |
| 长任务 | 一般无 | **轮询** progress |

上传按钮的 loading 只覆盖 **HTTP 上传**；索引进度由 **列表轮询** 展示——两个时间尺度不要混成一个 spinner。

### 3.3 初学者常问：为啥不用 Server Action 上传？

Server Actions **可以**传 `File`，但本系列第七～九篇已统一 **Client `fetch` + rewrites** 心智（流式、FormData 同路径）。初学先走通一种；熟练后再评估 Action 是否减少 Client 代码。

## 4. 后端：上传接口 + 内存任务状态机

演示什么：收文件、建任务、后台模拟 ingest。  
前置：`pip install python-multipart`（`UploadFile` 需要）。

`requirements.txt` 增加：

```text
python-multipart>=0.0.6
```

在 `main.py`（与第七～九篇 **同一 `app`**）追加：

```python
import uuid
from datetime import datetime, timezone
from enum import Enum

from fastapi import UploadFile, File, HTTPException, BackgroundTasks


class JobStatus(str, Enum):
    pending = "pending"
    running = "running"
    done = "done"
    failed = "failed"


_documents: dict[str, dict] = {}


class DocumentOut(BaseModel):
    id: str
    filename: str
    status: JobStatus
    progress: int
    error: str | None = None
    created_at: str


async def fake_ingest(document_id: str):
    doc = _documents[document_id]
    try:
        doc["status"] = JobStatus.running
        for p in range(0, 101, 10):
            doc["progress"] = p
            await asyncio.sleep(0.4)
        doc["status"] = JobStatus.done
        doc["progress"] = 100
    except Exception as e:
        doc["status"] = JobStatus.failed
        doc["error"] = str(e)


@app.post("/api/documents", response_model=DocumentOut, status_code=201)
async def upload_document(
    background_tasks: BackgroundTasks,
    file: UploadFile = File(...),
):
    if not file.filename:
        raise HTTPException(400, "缺少文件名")

    doc_id = str(uuid.uuid4())
    content = await file.read()
    _documents[doc_id] = {
        "id": doc_id,
        "filename": file.filename,
        "status": JobStatus.pending,
        "progress": 0,
        "error": None,
        "created_at": datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ"),
        "size": len(content),
    }
    background_tasks.add_task(fake_ingest, doc_id)
    return _documents[doc_id]


@app.get("/api/documents", response_model=list[DocumentOut])
def list_documents():
    return sorted(_documents.values(), key=lambda d: d["created_at"], reverse=True)


@app.get("/api/documents/{doc_id}", response_model=DocumentOut)
def get_document(doc_id: str):
    if doc_id not in _documents:
        raise HTTPException(404, "文档不存在")
    return _documents[doc_id]
```

**BackgroundTasks**（FastAPI 后台任务）：HTTP 响应返回**之后**再跑函数。  
通俗说：先回执「已收件」，再后台索引。

自测：

```bash
curl -X POST http://localhost:8000/api/documents -F "file=@./README.md"
curl http://localhost:8000/api/documents
```

### 4.3 内存字典 _documents 的局限（演示必知）

本篇 `_documents` 在 **进程内存**里：

| 行为 | 结果 |
|------|------|
| 刷新页面 | 列表仍在（GET 读内存） |
| 重启 uvicorn | **全部丢失** |
| 多 worker | 各 worker 各一份，不一致 |

向听众/面试官要主动说：「真项目用 PostgreSQL + 对象存储 + 任务队列。」第十篇教的是 **API 形状与前端轮询模式**，不是生产存储。

### 4.4 fake_ingest 进度为何用 0,10,…,100

每 0.4 秒 +10%，大约 4 秒走完——够你演示进度条动画，又不至于等太久。  
真 ingest 进度应来自「解析页数 / 已向量化 chunk 数」等可观测指标，而不是 `sleep` 猜的。

## 5. 前端：UploadForm 与上传三态

演示什么：`<input type="file">` + 提交三态。  
前置：`getApiRoot()`（第六篇）。

可选封装 `lib/documents.js`：

```javascript
// src/lib/documents.js
import { getApiRoot } from './api.js'

export function documentsUrl(id) {
  const base = `${getApiRoot()}/documents`
  return id ? `${base}/${id}` : base
}

export async function uploadDocument(file) {
  const form = new FormData()
  form.append('file', file)
  const res = await fetch(documentsUrl(), { method: 'POST', body: form })
  if (!res.ok) {
    const text = await res.text().catch(() => '')
    throw new Error(text || `HTTP ${res.status}`)
  }
  return res.json()
}

export async function listDocuments() {
  const res = await fetch(documentsUrl())
  if (!res.ok) throw new Error(`HTTP ${res.status}`)
  return res.json()
}
```

`UploadForm.js`：

```jsx
// src/components/UploadForm.js
'use client'

import { useState } from 'react'
import { uploadDocument } from '../lib/documents.js'

export default function UploadForm({ onUploaded }) {
  const [file, setFile] = useState(null)
  const [uploading, setUploading] = useState(false)
  const [error, setError] = useState(null)

  async function handleSubmit(e) {
    e.preventDefault()
    if (!file || uploading) return

    setUploading(true)
    setError(null)

    try {
      const doc = await uploadDocument(file)
      setFile(null)
      onUploaded?.(doc)
    } catch (err) {
      setError(err.message)
    } finally {
      setUploading(false)
    }
  }

  return (
    <form onSubmit={handleSubmit} style={{ marginBottom: 16 }}>
      <input
        type="file"
        accept=".md,.txt,.pdf"
        onChange={(e) => setFile(e.target.files?.[0] ?? null)}
        disabled={uploading}
      />
      <button
        type="submit"
        disabled={!file || uploading}
        style={{ marginLeft: 8 }}
      >
        {uploading ? '上传中…' : '上传并索引'}
      </button>
      {error && <p style={{ color: '#b91c1c' }}>{error}</p>}
    </form>
  )
}
```

**`accept`**：前端限制后缀；后端仍应白名单校验。

### 5.1 UploadForm 三态与用户体验

| 状态 | 按钮 | 文件 input | 用户理解 |
|------|------|------------|----------|
|  idle | 可点「上传」 | 可选文件 | 准备选文件 |
| `uploading` | 「上传中…」禁用 | 禁用 | 正在传文件到服务器 |
|  error | 恢复可点 | 可选 | 看红字重试 |

上传成功后 `setFile(null)` 清空 input——否则选同一文件 again 时 `onChange` 可能不触发（浏览器行为）。  
`onUploaded(doc)` 把 **201 返回的那一行**交给父组件插入列表，不必等 ingest 完成。

### 5.2 先错对对：把文件读成 base64 塞 JSON

```javascript
// ❌ 体积膨胀、无 multipart 语义
body: JSON.stringify({ name: file.name, data: await readAsDataURL(file) })

// ✅ 浏览器原生二进制通道
const form = new FormData()
form.append('file', file)
body: form
```

小文件 Demo 用 JSON base64「也能跑」，但不符合 REST 惯例，且大 PDF 会爆内存——**产品一律 FormData**。

## 6. 轮询任务状态：useEffect 与清理

**轮询**（Polling）：隔固定时间重复 `GET`，直到终态。  
通俗说：每隔几秒问「好了吗？」。

本篇用 **整表轮询**（文档少时最简单）——在 `documents/page.js` 内实现，不必单独 hook。

```javascript
useEffect(() => {
  let cancelled = false
  let timerId

  async function load() {
    try {
      const list = await listDocuments()
      if (cancelled) return
      setDocuments(list)
      const active = list.some(
        (d) => d.status === 'pending' || d.status === 'running'
      )
      if (active) timerId = setTimeout(load, 1500)
    } catch {
      /* 可加 error state */
    }
  }

  load()
  return () => {
    cancelled = true
    clearTimeout(timerId)
  }
}, [pollVersion])
```

**清理函数**：卸载或 `pollVersion` 变化时 `clearTimeout`，避免泄漏。  
**`pollVersion`**：曾全部 `done` 轮询已停时，**新上传**后 `setPollVersion(v => v + 1)` 重启轮询。

先错或对：

```javascript
// ❌ setInterval 永不 clear
// ✅ 终态停 + unmount 清理
```

### 6.1 为何 /documents 用 Client 整页

| 能力 | 需要 |
|------|------|
| `<input type="file">` | 浏览器事件 |
| `useState` 上传三态 | Client |
| `useEffect` 轮询 | Client |

第六篇占位用 Server 可以；**本篇替换为 `'use client'` 整页**最省事。进阶可拆 `DocumentsClient.js` 被 Server `page` import——初学不必。

### 6.2 轮询 effect 逐行走读

把 §6 代码块当成剧本：

| 行 | 作用 | 若删掉会怎样 |
|----|------|--------------|
| `let cancelled = false` | 防止卸载后 setState | 控制台 warning |
| `async function load()` | 一次拉全表 | — |
| `if (cancelled) return` | 卸载后不再更新 | 内存泄漏感 |
| `setDocuments(list)` | 刷新 UI | 进度不动 |
| `active = some(pending/running)` | 判断是否还要问 | done 后仍狂请求 |
| `setTimeout(load, 1500)` | 1.5s 后再问 | 进度不更新 |
| `return () => { cancelled=true; clearTimeout }` | 清理 | 泄漏定时器 |
| `[pollVersion]` 依赖 | 新上传重启 | 上传后进度卡住 |

**1.5 秒**是 Demo 折中：太快浪费请求，太慢显得卡。TanStack Query 的 `refetchInterval` 是同一思想的库封装（README 进阶）。

### 6.3 轮询 vs WebSocket 进度（了解）

| | 轮询（本篇） | WebSocket 推进度 |
|---|-------------|------------------|
| 实现 | `setTimeout` + GET | 后端主动 push |
| 适用 | 文档少、Demo | 大文件、多任务 |
| 缺点 | 空转请求 | 要维护连接 |

路线图 Celery + Redis 阶段常用 WebSocket 或 SSE 推任务事件；本篇轮询足够学会「**长任务要反馈**」的产品原则。

### 6.4 pollVersion 场景推演

| 场景 | 无 pollVersion | 有 pollVersion |
|------|----------------|----------------|
| 首次进页 | effect 跑，轮询到 done 停止 | 同左 |
| 全部 done 后再上传 | effect 不重启，新行 progress 不动 | 上传回调 `setPollVersion(v=>v+1)`，effect 重跑 |
| 用户快速连传两个文件 | 列表靠 POST 回执 + 下次轮询合并 | 每次上传 bump，尽快拉全表 |

`pollVersion` 不是 React 官方 API——只是 **「重启已停掉的 effect」** 的教程用技巧。TanStack Query 里等价于 `queryClient.invalidateQueries(['documents'])`。

### 6.5 与第三篇 useEffect 清理对照

[React（三）](../react/03.use-effect-data-fetching.md) 讲「拉用户列表要清理」；本篇轮询同一道理：

| | 用户列表 GET | 文档列表轮询 |
|---|-------------|--------------|
| 触发 | 进页一次 | 进页 + 有 active 任务时循环 |
| 清理 | `cancelled` 标志 | `cancelled` + `clearTimeout` |
| 重启 | 换路由 | `pollVersion` 变化 |

会写本篇 effect，就等于把 React（三）的清理模式用在了 **定时器** 上。

## 7. DocumentList：文档与进度 UI

```jsx
// src/components/DocumentList.js

const STATUS_LABEL = {
  pending: '排队中',
  running: '索引中',
  done: '已完成',
  failed: '失败',
}

export default function DocumentList({ documents }) {
  if (!documents.length) {
    return <p style={{ color: '#6b7280' }}>暂无文档，请先上传。</p>
  }

  return (
    <ul style={{ listStyle: 'none', padding: 0, margin: 0 }}>
      {documents.map((d) => (
        <li
          key={d.id}
          style={{
            border: '1px solid #e5e7eb',
            borderRadius: 8,
            padding: 12,
            marginBottom: 8,
          }}
        >
          <div style={{ display: 'flex', justifyContent: 'space-between' }}>
            <strong>{d.filename}</strong>
            <span>{STATUS_LABEL[d.status] ?? d.status}</span>
          </div>
          {(d.status === 'pending' || d.status === 'running') && (
            <div style={{ marginTop: 8 }}>
              <div
                style={{
                  height: 6,
                  background: '#e5e7eb',
                  borderRadius: 3,
                  overflow: 'hidden',
                }}
              >
                <div
                  style={{
                    width: `${d.progress}%`,
                    height: '100%',
                    background: '#2563eb',
                    transition: 'width 0.3s',
                  }}
                />
              </div>
              <span style={{ fontSize: 12, color: '#6b7280' }}>
                {d.progress}%
              </span>
            </div>
          )}
          {d.status === 'failed' && (
            <p style={{ color: '#b91c1c', fontSize: 13, marginTop: 8 }}>
              {d.error}
            </p>
          )}
        </li>
      ))}
    </ul>
  )
}
```

### 7.1 进度条与 status 的对应关系

| `status` | 进度条 | 百分比文案 | 用户该做什么 |
|----------|--------|------------|--------------|
| `pending` | 显示，可能 0% | 排队中 | 等 |
| `running` | 动画增长 | 索引中 x% | 等 |
| `done` | 隐藏 | 已完成 | 去 `/chat` |
| `failed` | 隐藏 | 失败 + error | 重传或联系管理员 |

`progress` 是 **0～100 整数**——后端 `fake_ingest` 每步 +10；真项目可能用「已处理 chunk / 总 chunk」换算。

### 7.2 列表为空时的文案

「暂无文档，请先上传。」引导用户 **先完成 ingest 再聊天**——与第十二篇演示脚本一致。  
不要空列表还链到 `/chat` 却不说明「可能没有检索结果」。

## 8. 综合实战：填满 /documents 页面

**阅读顺序**：§4～§7 完成后再写 `page.js`。

替换第六篇 Server 占位：

```jsx
// src/app/documents/page.js
'use client'

import { useEffect, useState } from 'react'
import Link from 'next/link'
import UploadForm from '../../components/UploadForm.js'
import DocumentList from '../../components/DocumentList.js'
import { listDocuments } from '../../lib/documents.js'

export default function DocumentsPage() {
  const [documents, setDocuments] = useState([])
  const [pollVersion, setPollVersion] = useState(0)

  useEffect(() => {
    let cancelled = false
    let timerId

    async function load() {
      try {
        const list = await listDocuments()
        if (cancelled) return
        setDocuments(list)
        const active = list.some(
          (d) => d.status === 'pending' || d.status === 'running'
        )
        if (active) timerId = setTimeout(load, 1500)
      } catch {
        /* 可选 setLoadError */
      }
    }

    load()
    return () => {
      cancelled = true
      clearTimeout(timerId)
    }
  }, [pollVersion])

  function handleUploaded(doc) {
    setDocuments((prev) => [doc, ...prev])
    setPollVersion((v) => v + 1)
  }

  return (
    <section style={{ maxWidth: 640 }}>
      <div
        style={{
          display: 'flex',
          justifyContent: 'space-between',
          alignItems: 'center',
        }}
      >
        <h1>文档与知识库</h1>
        <Link href="/chat">去问答 →</Link>
      </div>
      <p style={{ color: '#666', fontSize: 14 }}>
        上传后等待状态为「已完成」，再到{' '}
        <Link href="/chat">对话页</Link> 提问（Demo 建议）。
      </p>
      <UploadForm onUploaded={handleUploaded} />
      <DocumentList documents={documents} />
    </section>
  )
}
```

路由已由第六篇 `app/documents/page.js` 定义，**无需**改 `next.config.js`。

| 路由 | 职责 |
|------|------|
| `/documents` | 上传、索引进度 |
| `/chat` | 流式 + citations |

`SiteNav`（第六篇）已链两页。

| 步骤 | 操作 | 预期 | 若不对 |
|------|------|------|--------|
| 1 | `curl -F file=@README.md` | `201`，`pending` | §4、multipart |
| 2 | 打开 `/documents` | 空列表或历史行 | 路由、rewrites |
| 3 | 页面上传 `.md` | 立刻多一行，按钮恢复 | §5 三态 |
| 4 | 等待数秒 | `running`，进度条动 | §6 轮询 |

### 8.5 状态机走读（心里默念）

```text
pending  → 已创建任务，BackgroundTasks 尚未跑或刚排队
running  → fake_ingest 循环改 progress
done     → progress=100，轮询停止
failed   → 展示 error 文案（本篇模拟很少失败）
```

用户**不应**在 `pending` 很久——若一直 pending，查 `background_tasks.add_task` 是否调用。

### 8.6 闭卷口述

> 「浏览器 `FormData.append('file', file)` POST 到 `/api/documents`，不设 JSON Content-Type。FastAPI `UploadFile` 收文件，立刻 201 返回任务行，真正索引用 `BackgroundTasks` 异步跑。前端 `useEffect` 每 1.5s 拉列表，有 pending/running 就继续定时，否则 clearTimeout。新上传 bump `pollVersion` 重启 effect。」

### 8.7 分层排错

```text
层 1：curl -F 能 201？
  失败 → python-multipart、路由
  成功 ↓
层 2：页面 POST 能 201？
  失败 → FormData、rewrites、字段名 file
  成功 ↓
层 3：列表不刷新进度？
  失败 → listDocuments、轮询逻辑、pollVersion
  成功 ↓
层 4：done 仍每秒请求？
  → active 判断、忘记 clearTimeout
```

## 9. 常见陷阱与 FAQ

### 9.1 陷阱一：手写 Content-Type

`multipart` 需要 boundary；手写 `application/json` 会解析失败。

### 9.2 陷阱二：轮询不设终态

`done` 后仍每 1.5s 请求——见 §6 在 active 为 false 时不 `setTimeout`。

### 9.3 陷阱三：忘记 python-multipart

FastAPI `UploadFile` 报错 → `pip install python-multipart`。

### 9.4 陷阱四：上传与索引用一个 loading 锁死整页

上传结束即放开按钮；索引进度由列表轮询展示。

### 9.5 陷阱五：用 Server Action 传 File 却未搞清边界

Server Actions **可以**上传文件，但本篇与第七～九篇统一用 **Client `fetch` + rewrites**；避免混用两套心智。生产可再评估 Action。

### 9.6 陷阱六：Server `documents/page` 里写 `useEffect` 轮询

会报错——整页标 **`'use client'`** 或拆 Client 子组件。

### 9.7 FAQ

**Q：和 React（十）差在哪？**  
A：逻辑相同；页面在 `app/documents/page.js`，链调用 `next/link` 的 `href="/chat"`。

**Q：能用 React Query 吗？**  
A：`refetchInterval` 等价；见 README 进阶。

**Q：上传进度条？**  
A：需监听 upload 事件；本篇「上传中…」即可。

**Q：多文件？**  
A：多次 `form.append` + 后端 `List[UploadFile]`——扩展作业。

**Q：和第九篇引用里的 `docs/rag-intro.md` 啥关系？**  
A：第九篇是模拟数据；本篇上传后真文件名会出现在列表——演示时可上传同名 `.md` 形成「闭环故事」。

**Q：进度条能更平滑吗？**  
A：后端改 `sleep` 步长；或 WebSocket 推进度（路线图进阶）。

### 9.13 上传失败分层排错（初学者常遇）

| 现象 | 先查 | 常见原因 |
|------|------|----------|

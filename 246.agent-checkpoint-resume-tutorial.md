# AI Agent 工程（三十三）：Checkpoint 与 Resume

> Checkpoint 的目标不是保存一份聊天记录，而是保存能安全恢复执行的最小完整状态。

---

## 你会学到什么

- 设计 Checkpoint schema。
- 确定保存时机和恢复规则。
- 处理工具执行中的未知结果。
- 防止恢复后预算、版本和审批参数漂移。

## 它解决什么问题

任务可能因这些原因暂停：

- worker 重启。
- 等待 Human Approval。
- 外部依赖暂时不可用。
- 用户稍后补充信息。
- 长任务被调度到另一个 worker。

统一术语：**State、Transition、Checkpoint、Resume、Human Approval**。

## 最小示例

```python
from pydantic import BaseModel, Field


class Checkpoint(BaseModel):
    task_id: str
    state: str
    state_version: int
    workflow_version: str
    current_step: int
    remaining_steps: int
    evidence_ids: list[str] = Field(default_factory=list)
    completed_tool_calls: dict[str, dict] = Field(default_factory=dict)
    pending_approval_id: str | None = None
```

Resume：

```python
def resume(task_id: str) -> AgentTask:
    checkpoint = checkpoint_store.latest(task_id)
    verify_workflow_compatibility(checkpoint.workflow_version)
    verify_no_active_worker(task_id, checkpoint.state_version)
    return AgentTask.from_checkpoint(checkpoint)
```

## 工程化版本

### 保存时机

至少在这些边界保存：

1. 工具调用成功后。
2. 高风险工具执行前。
3. 进入 waiting_for_approval 前。
4. 完成一个计划步骤后。
5. 转后台任务前。

### 保存什么

```text
目标和用户上下文引用
State / Transition 版本
当前步骤与剩余预算
Evidence IDs
工具调用 ID、幂等键和结果摘要
Pending Human Approval ID
模型、prompt、tool、policy 版本
```

Secret 和大工具结果只保存受控引用。

### 工具执行边界

最危险的窗口：

```text
工具已成功 → Checkpoint 尚未保存 → 进程崩溃
```

写工具依赖幂等键；恢复时先查询 completed_tool_calls 或外部状态，再决定是否继续。

### Workflow 版本

旧任务默认使用原 workflow_version Resume。若必须迁移，提供显式 state migration，不要直接用新流程解释旧状态。

## 常见失败模式

- Checkpoint 只保存消息。
- 恢复时重置 max_steps。
- 工具成功但未记录幂等键。
- Human Approval 参数不在 Checkpoint。
- 新版本删除旧状态字段。
- 多个 worker 同时 Resume。

## 什么时候不要这么做

短小、无副作用、可安全从头重算的请求不需要持久化 Checkpoint。

不要保存整个模型上下文作为唯一恢复来源；它缺乏结构化一致性。

## 生产环境注意事项

- Checkpoint 写入幂等并带 state_version。
- 使用 lease 或锁避免重复 worker。
- Resume 前检查取消和过期状态。
- Approval 过期后不能继续执行。
- 预算、工具版本和权限上下文不自动刷新。
- 敏感字段加密或只保存引用。

## 如何观测和评测

指标：

- Checkpoint 写入成功率和延迟。
- Resume 成功率。
- 重复工具调用数。
- 旧 workflow 兼容失败数。
- 多 worker 冲突。
- waiting 状态平均持续时间。

故障演练：在每个工具调用前后强制终止 worker，验证恢复不会重复副作用。

## 和 RAG / 后端 / 前端的关系

- Evidence 保存 ID，RAG 内容从 Evidence Store 重载。
- 后端实现 Checkpoint、锁和迁移。
- 前端使用 task_id 查询当前 State。
- Human Approval 完成后发布 Resume 事件。

## 面试怎么讲

> Checkpoint 保存结构化 State、版本、预算、Evidence IDs、已完成工具调用和 pending approval，不是只存聊天。写工具使用幂等键覆盖“执行成功但 checkpoint 未保存”的窗口。Resume 使用原 workflow_version，先获取 lease 并检查状态版本、取消和审批有效性。

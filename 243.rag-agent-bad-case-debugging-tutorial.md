# AI Agent 工程（三十）：RAG Agent Bad Case Debugging

> Agentic RAG 的错误可能发生在规划、检索、工具、证据组合、引用或最终生成。没有完整轨迹时，团队只能反复改 prompt。

---

## 你会学到什么

- 建立分层根因分类。
- 从最终错误反向追踪执行轨迹。
- 保存可重放的 bad case。
- 用回归集验证修复没有制造新问题。

## 它解决什么问题

用户反馈“答案错了”，根因可能是：

```text
Query Planning 漏了子问题
Retrieval Tool 查错索引
ACL 过滤导致关键资料不可见
重排把旧版文档放到前面
业务工具返回缓存旧状态
Citation Verification 漏检
最终生成忽略证据
```

因此 **Bad Case Debugging** 必须串联 **Query Planning、Retrieval Tool、Citation Verification** 的全部数据。

## 最小示例

```json
{
  "case_id": "BC-20260722-01",
  "task": "为什么客户不能使用高级报表？",
  "query_plan": ["套餐要求", "当前角色权限"],
  "retrievals": [
    {
      "query_id": "q1",
      "index_version": "product-v8",
      "evidence_ids": ["doc-17"]
    }
  ],
  "tool_calls": [
    {
      "tool": "get_user_permissions",
      "resource_version": 31,
      "ok": true
    }
  ],
  "citation_verification": {
    "supported": false,
    "unsupported_claims": ["客户套餐不支持高级报表"]
  },
  "expected_root_cause": "generation_ignored_business_evidence"
}
```

## 工程化版本

### 根因分类

| 层级 | 常见根因 |
|---|---|
| Input | 用户目标或实体缺失 |
| Query Planning | 漏子问题、错数据源、重复查询 |
| Retrieval Tool | 索引、filter、ACL、top_k 错误 |
| Rerank | 旧版或无关文档排序过高 |
| Business Tool | 缓存旧状态、权限过滤错误 |
| Evidence | 去重、冲突、版本处理错误 |
| Citation Verification | 假引用、漏检 |
| Generation | 忽略证据、错误归纳 |
| Orchestration | 循环、预算、停止条件错误 |

### 反向追踪

```text
错误结论
  → 对应 claim 和引用
  → Citation Verification 结果
  → Evidence Store 原文
  → Retrieval Tool 输入输出
  → Query Planning
  → 用户原始任务和权限上下文
```

### 可重放

保存模型版本、prompt_version、tool_version、index_version、policy_version 和随机参数。敏感数据用脱敏快照。

### 一次只修一层

如果 Retrieval Tool 没召回，不要先改最终 prompt。先用固定 query 验证索引、filter 和重排。

## 常见失败模式

- 只保存最终答案。
- 所有 bad case 都归因“模型幻觉”。
- 同时修改 prompt、索引和 reranker，无法知道哪个有效。
- 回放使用了最新文档而非当时版本。
- 没有用户权限上下文。
- 修复一个案例后没有跑回归集。

## 什么时候不要这么做

不要把真实敏感数据直接复制进公共测试集。

随机不可复现的线上轨迹如果缺版本信息，应先补观测，不要假装已找到根因。

不要用单个 bad case 推导全局架构结论；先看同类错误频率。

## 生产环境注意事项

- Bad case 数据脱敏并受访问控制。
- 保存必要版本和决策摘要。
- 给每个修复绑定 root_cause 和 regression_cases。
- 区分数据问题、配置问题和代码问题。
- 修复上线后监控同类错误率。

## 如何观测和评测

维护回归表：

```text
case_id
root_cause
expected_tools
expected_evidence
forbidden_actions
expected_stop_reason
expected_claims
```

指标：

- 根因分类覆盖率。
- 可重放率。
- 平均定位时间。
- 修复后通过率。
- 新增回归数。
- 同类问题复发率。

## 和 RAG / 后端 / 前端的关系

- RAG 暴露查询、filter、召回和重排版本。
- 后端保存完整 trace 和 Policy 决策。
- 前端反馈入口附带 trace_id。
- Citation Verification 结果直接进入 bad case。

## 面试怎么讲

> 我会从错误 claim 反向追踪引用、Evidence Store、Retrieval Tool、Query Planning 和原始输入，按层定位。bad case 保存模型、prompt、tool、index 和 policy 版本，修复时一次只改一层，并把案例加入回归集。没有完整轨迹时先补观测，不先猜 prompt。

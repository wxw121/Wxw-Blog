# AI Agent 工程（九）：工具错误、超时与重试

> Agent 调用的系统越多，失败越正常。可靠性不来自“多试几次”，而来自错误分类、超时预算、受控重试和明确降级。

---

## 你会学到什么

- 区分可重试和不可重试错误。
- 为模型、工具和整条任务设置分层超时。
- 使用指数退避和重试预算。
- 防止模型重试与基础设施重试叠加。

## 它解决什么问题

常见工具失败包括：

| 错误 | 是否重试 | 原因 |
|---|---|---|
| 网络瞬时超时 | 可以 | 外部依赖短暂不可用 |
| 429 限流 | 可以 | 等待后可能恢复 |
| 参数格式错误 | 修正后再试 | 原参数重试不会成功 |
| 权限拒绝 | 不可以 | 重试不能获得权限 |
| 资源不存在 | 通常不可以 | 需要换查询目标 |
| 写入结果未知 | 谨慎 | 可能已经成功，需幂等查询 |

如果所有错误都让模型自行判断，很容易产生无限重试或重复写入。

## 最小示例

```python
import asyncio
from dataclasses import dataclass


@dataclass(frozen=True)
class ToolFailure(Exception):
    code: str
    retryable: bool


async def call_with_timeout() -> dict:
    try:
        async with asyncio.timeout(3):
            return await query_ticket_service()
    except TimeoutError as exc:
        raise ToolFailure("ticket_service_timeout", True) from exc
```

带有限重试：

```python
async def call_with_retry(max_attempts: int = 3) -> dict:
    for attempt in range(1, max_attempts + 1):
        try:
            return await call_with_timeout()
        except ToolFailure as exc:
            if not exc.retryable or attempt == max_attempts:
                raise
            await asyncio.sleep(0.2 * (2 ** (attempt - 1)))

    raise RuntimeError("unreachable")
```

## 工程化版本

### 三层超时

```text
单次工具超时 < 单步超时 < 整体任务超时
```

示例：

| 层级 | 超时 |
|---|---|
| 单次 HTTP 工具 | 3 秒 |
| Agent 单步 | 8 秒 |
| 整体同步请求 | 30 秒 |
| 后台长任务 | 10 分钟 |

整体预算耗尽后，不能因为某个工具还有重试次数就继续。

### 统一错误分类

```python
from typing import Literal


FailureKind = Literal[
    "invalid_arguments",
    "permission_denied",
    "not_found",
    "rate_limited",
    "timeout",
    "dependency_unavailable",
    "conflict",
    "unknown_result",
]


def retryable(kind: FailureKind) -> bool:
    return kind in {
        "rate_limited",
        "timeout",
        "dependency_unavailable",
    }
```

### 防止双重重试

如果 HTTP SDK 重试 3 次、工具包装器重试 3 次、Agent 又重试 3 次，最坏会执行 27 次。

应明确每层职责：

- SDK：只处理连接级瞬时错误，最多 1 次。
- 工具层：根据错误码重试，最多 2–3 次。
- Agent：工具最终失败后换方案或停止，不重复相同调用。

## 常见失败模式

- 没有超时，任务永久停在 running。
- 所有异常都标记 retryable。
- 重试时修改写操作参数。
- 没有 jitter，大量任务同时重试。
- 重试日志缺少 attempt 和 elapsed time。
- 工具失败后模型换个说法调用同一个工具，绕过重试上限。

## 什么时候不要这么做

权限拒绝、参数非法和确定性业务冲突不应自动重试。

写操作返回“结果未知”时，不要直接再次写入；先用幂等键查询原请求状态。

如果整体任务已经超过用户可接受延迟，应转后台任务或直接降级，不要继续同步重试。

## 生产环境注意事项

每个任务维护重试预算：

```json
{
  "max_total_tool_calls": 12,
  "max_attempts_per_tool": 3,
  "max_elapsed_ms": 30000,
  "consecutive_failure_limit": 2
}
```

模型看到的错误应该简洁：

```json
{
  "code": "ticket_service_timeout",
  "retryable": false,
  "message": "工单服务暂时不可用，请使用已有证据回答或转人工。"
}
```

这里 `retryable=false` 表示基础设施已经耗尽自动重试，不允许模型再次调用。

## 如何观测和评测

记录每次 attempt：

- tool_name。
- attempt_index。
- failure_kind。
- timeout_ms。
- elapsed_ms。
- next_backoff_ms。
- final_outcome。

关注重试放大率：真实业务请求数 / Agent 工具调用意图数。这个比例持续升高通常意味着依赖不稳定或重试层叠。

## 和 RAG / 后端 / 前端的关系

- RAG 超时可降级为已有上下文回答或明确说明资料不足。
- 后端负责重试和超时，模型不负责 sleep。
- 前端区分“正在重试”“已转后台”“需要人工处理”。
- SSE 需要发送 heartbeat，避免长工具调用被误认为连接断开。

## 面试怎么讲

> 我会先把错误分类为参数、权限、业务冲突、限流、超时和未知结果，只对瞬时错误重试。超时分工具、单步和整体任务三层，并设置总重试预算，防止 SDK、工具层和 Agent 三层重试相乘。写操作结果未知时先按幂等键查状态，不直接重放。

# AI Agent 工程（十四）：ReAct 模式

> ReAct 的核心价值不是要求模型输出长篇思维过程，而是让“下一步行动”建立在最新 observation 上，并且每一步都可验证。

---

## 你会学到什么

- 理解 ReAct 的 Reason + Act 结构。
- 使用简短 decision summary 替代隐藏思维日志。
- 控制每轮只执行一个可验证动作。
- 识别 ReAct 容易产生的循环和成本问题。

## 它解决什么问题

复杂任务很难一次生成完整正确答案。ReAct 允许模型：

```text
理解当前缺什么
  → 选择一个动作
  → 得到 observation
  → 根据新证据调整下一步
```

例如排查“客户为什么无法查看高级报表”：

1. 查询客户套餐。
2. Observation：套餐支持高级报表。
3. 查询用户角色。
4. Observation：角色缺少 `report:read`。
5. 根据证据生成原因和处理建议。

## 最小示例

```python
from pydantic import BaseModel, Field
from typing import Literal


class ReActDecision(BaseModel):
    action: Literal["call_tool", "finish"]
    decision_summary: str = Field(max_length=300)
    tool_name: str | None = None
    arguments: dict = Field(default_factory=dict)
    final_answer: str | None = None
```

每轮只要求模型说明可审计的决策摘要：

```json
{
  "action": "call_tool",
  "decision_summary": "已确认套餐支持报表，还需要检查用户角色权限。",
  "tool_name": "get_user_permissions",
  "arguments": {
    "user_id": "U-18"
  }
}
```

不要把模型不可验证的长篇内部推理保存为业务事实。

## 工程化版本

ReAct 执行器仍然使用 Agent Loop：

```python
def react_step(state: AgentState) -> AgentState:
    decision = planner.next_action(
        goal=state.goal,
        evidence=state.evidence,
        allowed_tools=state.allowed_tools,
        remaining_steps=state.max_steps - state.step,
    )

    validate_decision(decision, state.allowed_tools)

    if decision.action == "finish":
        state.final_answer = decision.final_answer
        state.stop_reason = "completed"
        return state

    result = tool_registry.execute(
        decision.tool_name,
        decision.arguments,
        context=state.user_context,
    )
    state.record(decision, result)
    return state
```

### 给模型最小 observation

不要把完整日志或数据库行回传。只给决策所需字段和证据身份。

### 检测无进展

如果最近两步调用同一工具、参数相同、结果相同，应停止或换策略。

```python
def no_progress(steps: list[Step]) -> bool:
    if len(steps) < 2:
        return False
    return steps[-1].fingerprint == steps[-2].fingerprint
```

## 常见失败模式

- 把长篇思维过程当作真实审计记录。
- 每轮执行多个工具，失败时无法定位。
- 工具结果不完整，却让模型假装任务完成。
- 重复调用同一工具获取相同结果。
- 在简单任务中使用 ReAct，增加延迟和成本。
- decision_summary 包含未验证事实。

## 什么时候不要这么做

固定步骤不需要 ReAct；使用 Workflow 更可靠。

只需一次检索和回答时，普通 RAG 更简单。

高风险写操作不能因为 ReAct“推理充分”就跳过 Policy 和人工确认。

## 生产环境注意事项

- 单轮只允许一个工具动作。
- decision schema 必须校验。
- 工具结果写入 state 后再开始下一轮。
- 为重复调用、无证据结论和最大步数建立守卫。
- 不把隐藏思维作为合规审计依据。

## 如何观测和评测

评测轨迹：

| 维度 | 问题 |
|---|---|
| Action relevance | 这一步工具是否有助于目标？ |
| Evidence use | 下一步是否使用了上一条 observation？ |
| Progress | 是否减少了未知信息？ |
| Stop quality | 是否在证据足够时停止？ |
| Safety | 是否遵守权限和审批？ |

## 和 RAG / 后端 / 前端的关系

- RAG 是 ReAct 可选择的一个只读动作。
- 后端校验每次 Action。
- 前端展示 decision summary 和工具状态，不展示私有推理。
- Workflow 可在确定节点调用一个有限步数的 ReAct 子流程。

## 面试怎么讲

> ReAct 让决策和 observation 交替，但生产实现不依赖模型输出长篇思维过程。我会让模型返回结构化 action 和简短 decision summary，每轮只执行一个工具，并检测重复调用和无进展。权限、参数和停止条件仍由编排器控制。

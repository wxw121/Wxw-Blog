# AI Agent 工程（十）：幂等 Agent 工具

> Agent 会因为超时、网络中断、任务恢复和模型重规划重复调用工具。写操作如果没有幂等设计，重试会变成重复扣款、重复工单或重复通知。

---

## 你会学到什么

- 理解 Agent 场景为什么更需要幂等。
- 设计稳定的 idempotency key。
- 区分重复请求、冲突请求和未知结果。
- 在 Checkpoint 恢复后复用原调用结果。

## 它解决什么问题

典型故障：

```text
Agent 调用 create_ticket
  → 工单服务成功创建
  → 返回响应时网络超时
  → Agent 认为失败
  → 重试 create_ticket
  → 创建第二张工单
```

幂等目标是：相同业务意图重复提交，只产生一次副作用，并返回同一个结果。

## 最小示例

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class CreateTicketInput:
    customer_id: str
    subject: str
    content: str
    idempotency_key: str


def create_ticket(value: CreateTicketInput) -> dict:
    previous = idempotency_store.get(value.idempotency_key)
    if previous is not None:
        return previous

    ticket = ticket_repository.create(
        customer_id=value.customer_id,
        subject=value.subject,
        content=value.content,
    )
    result = {"ticket_id": ticket.id, "created": True}
    idempotency_store.save(value.idempotency_key, result)
    return result
```

真实实现需要事务或唯一约束，避免两个并发请求同时通过 `get`。

## 工程化版本

### Key 从业务意图生成

不要每次重试生成新 UUID。推荐：

```text
task_id + step_id + tool_name + approved_action_version
```

例如：

```text
task-901:step-04:create_ticket:v1
```

### 数据库唯一约束

```sql
CREATE TABLE tool_idempotency (
    idempotency_key VARCHAR(200) PRIMARY KEY,
    request_hash VARCHAR(64) NOT NULL,
    status VARCHAR(20) NOT NULL,
    response_json JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

执行流程：

1. 插入 `processing` 记录。
2. 主键冲突时读取原记录。
3. 比较 request_hash。
4. 相同请求返回旧结果。
5. 不同请求使用同一 key 时返回 conflict。

```python
def handle_duplicate(
    existing: IdempotencyRecord,
    request_hash: str,
) -> dict:
    if existing.request_hash != request_hash:
        raise ValueError("idempotency_key_reused_with_different_request")
    if existing.status == "completed":
        return existing.response_json
    raise RuntimeError("original_request_still_processing")
```

### 恢复时复用结果

Checkpoint 应保存 tool_call_id、idempotency_key 和结果摘要。任务恢复后先查询旧调用，不能让模型重新生成新 key。

## 常见失败模式

- key 每次请求随机生成，重试无法命中。
- 只在 Agent 内存里去重，进程重启后失效。
- 相同 key 可以绑定不同参数。
- 只保存成功结果，不保存处理中状态。
- 工具成功但保存幂等记录失败。
- 审批后重新生成动作参数和 key。

## 什么时候不要这么做

纯只读查询通常天然幂等，不需要复杂幂等表；但仍要限制重复查询和成本。

不能简单通过“查询是否已经存在相似工单”代替幂等键，因为相似不等于同一业务意图。

## 生产环境注意事项

对外部系统：

- 优先使用对方提供的 idempotency key。
- 如果对方不支持，建立本地 outbox 和状态查询。
- 结果未知时先 query status。
- 为幂等记录设置符合业务审计周期的保留时间。

人工确认动作必须把 key 与批准内容绑定：

```json
{
  "approval_id": "approval-88",
  "action_version": 1,
  "idempotency_key": "task-901:step-04:create_refund:v1"
}
```

## 如何观测和评测

监控：

- 幂等命中次数。
- key 冲突次数。
- processing 超时记录。
- 未知结果查询次数。
- 重复副作用事故数。

测试必须覆盖并发提交、超时后重试、任务恢复和相同 key 不同参数。

## 和 RAG / 后端 / 前端的关系

- RAG 检索不需要业务幂等，但要缓存和去重。
- 后端数据库或外部 API 承担最终幂等保证。
- 前端重复点击确认时复用同一个 approval_id 和 idempotency_key。
- SSE 重连不能重新触发写工具。

## 面试怎么讲

> Agent 会因重试和恢复重复调用写工具，所以幂等键要从稳定业务意图生成，不能每次随机。数据库用唯一约束保存 processing/completed 状态，并校验 request_hash。任务恢复和前端重复确认都复用原 key；结果未知时先查状态，不直接重放。
